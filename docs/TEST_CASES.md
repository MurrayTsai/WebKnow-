# 测试用例与评估记录

## 1. 测试目标

本测试集用于验证 WebKnow 的核心产品能力：

1. **知识库命中**：问题能否被 Dify Knowledge 直接召回并回答。
2. **准实时补充**：知识库不足时，系统能否读取 sitemap、定位相关文章 URL，并通过 Jina Reader 抓取文章正文回答。
3. **无答案拒答**：知识库和站内页面都没有内容时，系统能否拒绝编造。
4. **相关性判断**：LLM 判断节点能否区分“直接相关、泛相关、不相关”。
5. **合规边界**：涉及账号封禁、风控、平台规则时，是否避免输出规避建议。
6. **可追溯性**：回答是否基于知识库或站内页面，并尽量提供来源。

### 5.1 主要问题归类

1. **知识库召回不稳定**：部分产品定义、流程、指标类知识块没有被稳定命中，例如 T01、T05、T14。
2. **判断节点相关性误判**：部分问题被过严判断为无答案，或把泛相关内容误判为可回答，需要继续优化 YES/NO 判断 Prompt。
3. **sitemap 实时补充分支稳定性不足**：T17 暴露出 sitemap URL 选择失败，需要加强标题关键词匹配和 URL 清洗。
4. **回答结构不够贴合用户问题**：T20 命中文章但没有先回答“如何识别核心用户”的总原则，需要让回答节点先直接回答，再展开文章细节。
5. **合规风险**：T29 为 P0，账号违规、风控、平台限制类问题必须禁止输出规避建议，只提供官方规则、申诉和客服路径。

## 6. 测试结论

本轮 30 条测试中，21 条通过，5 条需优化，4 条未通过。整体看，WebKnow 的三条核心路径已经具备可用性：

1. **知识库命中路径可用**：项目介绍、工具作用、流程机制等问题可以直接由知识库回答。
2. **sitemap + Jina 实时补充路径可用**：AI Agent、RAG、Prompt、HTML5 等站内文章类问题能够通过实时补充回答。
3. **无答案拒答路径可用**：无关问题可以进入拒答分支，避免编造。

下一阶段优化重点：

- 提高产品定义、流程链路、评估指标类知识块召回权重。
- 将 TopK 调整到 4-5，并测试 Hybrid 检索和关键词同义触发。
- 优化 LLM 判断节点，避免泛相关误判和误拒答。
- 强化 sitemap URL 选择规则，优先匹配文章标题和正文相关度。
- 对平台违规、账号封禁、风控类问题加入硬性合规边界。
