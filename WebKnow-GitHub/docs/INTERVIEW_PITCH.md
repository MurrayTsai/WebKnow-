# 面试讲解稿

## 1. 项目一句话介绍

WebKnow 是一个准实时网站知识库问答助手，基于 Dify Chatflow 实现“知识库 RAG 优先 + sitemap 实时补充 + Jina Reader 网页读取”的多层问答流程。

---

## 2. 为什么做这个项目？

普通网站内容很多，但用户很难快速找到答案。传统站内搜索只能做关键词匹配，普通 RAG 又依赖提前导入知识库。如果网站更新了新文章，或者知识库漏抓了页面，机器人就容易回答不了或者产生幻觉。

所以我设计了一个多层兜底流程：

1. 优先查知识库，保证速度和稳定性。
2. 知识库没有答案时读取 sitemap，找到站内相关文章。
3. 再用 Jina Reader 实时读取文章正文。
4. 如果站内也没有内容，就明确拒答。

---

## 3. 核心流程怎么设计？

用户提问后，系统先进入知识库检索节点。检索结果不会直接拿去回答，而是先经过一个 LLM 判断节点，判断知识库内容是否足够回答。

如果足够，就进入知识库回答节点。

如果不足，就读取网站 sitemap，并让 LLM 从 sitemap 中选择最相关的文章详情页 URL。之后使用 Jina Reader 获取页面正文，再判断正文是否足够回答。如果足够，就基于正文回答；如果不足，就拒答。

---

## 4. 项目有什么亮点？

第一个亮点是它不是单纯的 RAG，而是知识库优先加实时补充。这样既保证常见问题回答速度，也能覆盖知识库漏抓或新增文章。

第二个亮点是我设计了多个判断节点，包括知识库是否足够、URL 是否有效、文章正文是否足够。这样可以减少无依据回答。

第三个亮点是我做了测试集和 badcase 分析。测试覆盖知识库命中、实时补充、跨概念对比、无答案拒答和合规风险。

---

## 5. 发现了什么问题？

测试中发现了五类典型问题：

1. 判断过严：例如“VPN 和工具怎么用”，知识库里有相关操作内容，但被判断为无答案。
2. 判断过宽：例如“AI 编造事实怎么办”，只命中了泛 AI 内容，却被误判为可以回答。
3. 合规风险：例如 TikTok 违规提示问题，模型可能给出规避平台规则的建议。

我针对这些问题优化了判断 Prompt 和回答 Prompt，加入了操作类判断规则、核心概念匹配规则和平台合规边界。

---

## 6. 这个项目体现了哪些 AI 产品能力？

- RAG 产品流程设计
- Dify Chatflow 编排
- 知识库与实时网页结合
- Prompt 约束设计
- badcase 分析
- 测试评估设计
- AI 回答边界和合规意识

## 7. 简历表述

WebKnow：准实时网站知识库问答助手

- 基于 Dify Chatflow 设计网站问答助手，构建“知识库 RAG 优先 + sitemap 实时补充 + Jina Reader 网页读取”的多层回答流程。
- 设计 LLM 判断节点，对知识库检索结果、sitemap URL、实时网页正文进行有效性判断，实现知识库命中、实时补充和无答案拒答三种路径。
- 使用 Jina Reader 将站内网页转换为 Markdown，并通过 sitemap 定位相关文章，实现知识库未命中场景下的准实时站内内容补充。
- 设计 30 条测试用例，覆盖知识库命中、跨概念对比、站内实时补充、操作类问答、合规风险和无答案拒答等场景。
- 通过 badcase 分析优化 Prompt 规则，降低泛相关误判、操作类误拒答和平台规则规避类回答风险。
